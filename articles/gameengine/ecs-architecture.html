Entity-Component-System Architecture

Oleh Admin · 28 Januari 2025 · ⏱ 12 menit baca
#ecs #game-engine #data-oriented #cpp #architecture

Entity-Component-System (ECS) adalah pola arsitektur yang mendominasi game engine modern — dari Unity (DOTS), Unreal (Mass Entity), hingga engine indie seperti Bevy dan Flecs. ECS bukan sekadar tren; ia hadir karena menjawab masalah nyata: skalabilitas, performa cache, dan fleksibilitas desain yang tidak bisa diberikan oleh OOP konvensional.

💡
Artikel ini membahas: konsep ECS, masalah yang ia selesaikan, implementasi minimal di C++, perbandingan dengan pendekatan OOP, dan layout memori cache-friendly untuk performa maksimal.

Masalah dengan OOP Konvensional

Dalam game engine berbasis OOP klasik, kita biasanya membuat hierarki kelas seperti ini:

classDiagram class GameObject { +position: vec3 +update() +render() } class PhysicsObject { +velocity: vec3 +mass: float +applyForce() } class RenderableObject { +mesh: Mesh* +material: Material* +draw() } class Enemy { +health: int +attack() +die() } GameObject <|-- PhysicsObject PhysicsObject <|-- RenderableObject RenderableObject <|-- Enemy

Masalah mulai muncul saat game semakin kompleks. Bagaimana jika kita ingin objek yang bisa di-render tapi tidak punya fisika? Atau objek yang invisible tapi tetap punya collision? Inheritance tree menjadi tidak fleksibel — setiap kombinasi fitur membutuhkan subclass baru. Ini disebut the diamond problem dan inheritance hell.

⚠️
Inheritance Hell: Semakin dalam hierarki inheritance, semakin sulit untuk mengubah behavior satu objek tanpa memengaruhi yang lain. Juga, cache miss meningkat karena objek-objek tersebar acak di heap.

Implementasi Minimal di C++

Implementasi ECS yang benar-benar produksi (seperti EnTT atau Flecs) sangat kompleks. Berikut implementasi minimal yang menjelaskan konsep intinya:

Component Storage — Array of Structs

#include <cstdint>
#include <unordered_map>
#include <vector>
#include <typeindex>
#include <memory>
#include <functional>

using EntityID = uint32_t;

// Base storage interface
struct IComponentArray {
    virtual ~IComponentArray() = default;
    virtual void remove(EntityID id) = 0;
};

// Typed dense component storage
template<typename T>
struct ComponentArray : IComponentArray {
    std::vector<T>         components;     // dense array
    std::vector<EntityID>  entities;       // parallel: entity per slot
    std::unordered_map<EntityID, size_t> indexMap; // entity → slot

    void insert(EntityID id, T component) {
        indexMap[id] = components.size();
        components.push_back(component);
        entities.push_back(id);
    }

    T* get(EntityID id) {
        auto it = indexMap.find(id);
        if (it == indexMap.end()) return nullptr;
        return &components[it->second];
    }

    void remove(EntityID id) override {
        auto it = indexMap.find(id);
        if (it == indexMap.end()) return;

        // Swap with last to keep array dense
        size_t idx     = it->second;
        size_t last    = components.size() - 1;
        EntityID lastId = entities[last];

        components[idx] = components[last];
        entities[idx]   = lastId;
        indexMap[lastId] = idx;

        components.pop_back();
        entities.pop_back();
        indexMap.erase(id);
    }
};

World — Registry Utama

struct World {
    EntityID nextId = 1;

    // Component storages keyed by type
    std::unordered_map<std::type_index,
        std::unique_ptr<IComponentArray>> storages;

    EntityID createEntity() { return nextId++; }

    void destroyEntity(EntityID id) {
        for (auto& [_, storage] : storages)
            storage->remove(id);
    }

    template<typename T>
    void addComponent(EntityID id, T component) {
        getStorage<T>().insert(id, component);
    }

    template<typename T>
    T* getComponent(EntityID id) {
        return getStorage<T>().get(id);
    }

    // Iterate all entities that have BOTH T and U
    template<typename T, typename U, typename Fn>
    void forEach(Fn&& fn) {
        auto& ta = getStorage<T>();
        for (size_t i = 0; i < ta.entities.size(); ++i) {
            EntityID id = ta.entities[i];
            U* u = getComponent<U>(id);
            if (u) fn(id, ta.components[i], *u);
        }
    }

private:
    template<typename T>
    ComponentArray<T>& getStorage() {
        auto key = std::type_index(typeid(T));
        if (!storages.count(key))
            storages[key] = std::make_unique<ComponentArray<T>>();
        return static_cast<ComponentArray<T>&>(*storages[key]);
    }
};

Mendefinisikan Komponen dan Sistem

// Pure data components
struct Position { float x, y, z; };
struct Velocity { float vx, vy, vz; };
struct Health   { int current, max; };
struct Renderable {
    uint32_t meshId;
    uint32_t materialId;
    bool     visible = true;
};

// Systems — pure functions
void movementSystem(World& world, float dt) {
    world.forEach<Position, Velocity>([dt](EntityID, Position& pos, Velocity& vel) {
        pos.x += vel.vx * dt;
        pos.y += vel.vy * dt;
        pos.z += vel.vz * dt;
    });
}

void healthSystem(World& world) {
    world.forEach<Health, Renderable>([](EntityID, Health& hp, Renderable& r) {
        // Hide entity if dead
        if (hp.current <= 0) r.visible = false;
    });
}

// Main game loop
int main() {
    World world;

    // Create player
    EntityID player = world.createEntity();
    world.addComponent(player, Position{ 0, 0, 0 });
    world.addComponent(player, Velocity{ 5, 0, 0 });
    world.addComponent(player, Health{ 100, 100 });
    world.addComponent(player, Renderable{ 1, 2 });

    // Create 1000 enemies effortlessly
    for (int i = 0; i < 1000; ++i) {
        EntityID e = world.createEntity();
        world.addComponent(e, Position{ (float)i * 2.f, 0, 0 });
        world.addComponent(e, Velocity{ -1.f, 0, 0 });
        world.addComponent(e, Health{ 50, 50 });
        world.addComponent(e, Renderable{ 3, 4 });
    }

    const float dt = 1.0f / 60.f;
    while (true) {
        movementSystem(world, dt);
        healthSystem(world);
        // renderSystem(world, renderer);
    }
}

Cache-Friendly Layout: SoA vs AoS

Salah satu keunggulan utama ECS adalah kemampuan untuk menyimpan komponen dalam Structure of Arrays (SoA) alih-alih Array of Structures (AoS).

PendekatanLayout MemoriCache Behavior
AoS (OOP klasik) [pos|vel|hp][pos|vel|hp][pos|vel|hp]... ❌ Saat iterasi position, data vel dan hp ikut di-load (waste)
SoA (ECS) [pos][pos][pos]... [vel][vel][vel]... [hp][hp]... ✅ Movement system hanya load position + velocity — 100% cache hit 
// AoS — BURUK untuk iterasi sistem
struct GameObjectAoS {
    Position   pos;      // 12 bytes
    Velocity   vel;      // 12 bytes
    Health     hp;       // 8 bytes
    Renderable render;   // 12 bytes
    // Total: 44 bytes per object — banyak terbuang saat iterasi pos saja
};
std::vector<GameObjectAoS> objects; // [44][44][44]...

// SoA — BAIK: movement system hanya sentuh 24 bytes (pos+vel) per entity
struct WorldSoA {
    std::vector<Position>   positions;   // [12][12][12]...
    std::vector<Velocity>   velocities;  // [12][12][12]...
    std::vector<Health>     healths;     // hanya dimuat oleh combat system
    std::vector<Renderable> renderables; // hanya dimuat oleh render system
};
// Cache line 64 bytes = 5 Position + 1 byte padding → hampir sempurna
🎯
Benchmark nyata: Pada 100.000 entitas, movement system dengan SoA bisa 5–10× lebih cepat dari AoS karena cache hit rate yang jauh lebih tinggi. Ini bukan premature optimization — ini fundamental design.

Checklist Implementasi ECS

  • Definisikan Entity sebagai ID integer (uint32_t)
  • Buat ComponentArray dengan dense storage + sparse map
  • Buat World sebagai registry komponent per type
  • Tulis System sebagai free function atau functor
  • Implementasi forEach<T, U> untuk iterasi multi-komponen
  • Tambahkan Archetype untuk pengelompokan entity sejenis
  • Implementasi Query cache untuk menghindari iterasi ulang
  • Tambahkan Event system untuk komunikasi antar system
  • Implementasi Entity handle versioning (generation counter) untuk mencegah dangling ID
  • Profiling: GPU-assisted instancing dari component data

Kesimpulan

ECS bukan silver bullet, tapi ia adalah alat yang tepat untuk game engine yang perlu mengelola ribuan hingga jutaan entitas dengan performa tinggi. Kunci suksesnya: data terpisah dari logika, layout memori yang cache-friendly, dan sistem yang bisa dijalankan secara independen dan paralel.
